# AI-SPEC — Phase 3: Safe Tool Completion

> AI design contract generated by `$gsd-ai-integration-phase`. Consumed by the planner and eval auditor.
> This phase intentionally selects the project's native Rust protocol/core runtime instead of adding an external agent framework.

---

## 1. System Classification

**System Type:** Autonomous Agent + Code Automation (single model, local bounded tools)

**Description:**
MiniMax Codex is a local CLI agent for non-programmers and developers. In this phase the pinned Provider may request one of eight strict local tools, but a deterministic Rust coordinator—not the model—owns identity, approval, policy, persistence, execution order, recovery, and the decision to continue. Good behavior means the model selects a useful allowed tool with valid arguments, the runtime makes every external effect understandable and recoverable, and the final answer is based on a durable structured result rather than an invented observation.

**Critical Failure Modes:**

1. A tool executes without the approval required by the active process mode, or `full-access` bypasses a hard safety gate.
2. A duplicate/invalid call ID or Provider history mismatch associates a result with the wrong request or executes an effect twice.
3. An interruption after a possible side effect is reported as success/cancelled or is replayed automatically instead of becoming `indeterminate`.
4. A model escapes the project, accesses secret/runtime metadata, launches an arbitrary shell/interpreter, or requests a destructive/install/network operation through schema ambiguity.
5. A tool result reaches the next Provider round before it is durable, or raw stderr/credentials/private reasoning enter the journal or model context.

## 1b. Domain Context

**Industry Vertical:** Local developer tooling / coding agents

**User Population:** Primarily non-programmers who describe an outcome, plus maintainers who inspect exact CLI behavior and recovery evidence on Windows or Linux.

**Stakes Level:** High

**Output Consequence:** The model's structured request can read project material, change source files, or launch a bounded diagnostic process. A wrong decision can corrupt work, expose credentials, mislead the user about what happened, or leave a process running.

### What Domain Experts Evaluate Against

| Dimension | Good (expert accepts) | Bad (expert flags) | Stakes | Source |
|-----------|-----------------------|--------------------|--------|--------|
| Call identity | One Provider-issued ID remains unchanged from streamed call through durable result and follow-up request. | Missing, regenerated, duplicated, or cross-wired ID. | Critical | Provider function-calling contracts + TOOL-01 |
| Approval clarity | Prompt shows normalized action, exact arguments, project scope, and effect class; explicit approval is bound to one ID. | Vague “allow?” prompt, sticky grant, pressure, or approval applied to another call. | Critical | SPEC AC-06/17/18 |
| Tool choice and schema | The model selects one known tool and produces a strict JSON object that passes the published schema. | Hallucinated tool, extra fields, command string, ambiguous path, or malformed JSON. | High | TOOL-04/05 |
| Effect containment | Every path/effect stays in the canonical project and inside the finite non-destructive action set. | Symlink/reparse escape, secret access, deletion, Git mutation, install, network, interpreter, or background work. | Critical | TOOL-05 |
| Recovery honesty | An uncertain post-start outcome is visibly `indeterminate`, never replayed, and blocks false completion. | Guessed success, guessed cancellation, or automatic re-execution. | Critical | master SPEC failure matrix |
| Result usefulness | Bounded structured output includes status/code and enough safe evidence for the model to continue. | Truncated without signal, unbounded output, raw stderr/secret, or prose that hides failure. | High | SPEC AC-08..15 |

### Known Failure Modes in This Domain

- Models invent close-but-unknown tool names or fields; permissive deserialization turns that hallucination into an unintended operation.
- Streaming APIs split a call name/arguments across chunks; acting before the terminal argument event produces malformed or partial execution.
- Agent loops resend the same call after timeout/restart and duplicate a write or process effect.
- “Full access” is interpreted as “no safety”, even though it should mean only “do not ask again in this process”.
- Shell wrappers make command injection and platform differences invisible; a denylist misses interpreters, aliases, nested package scripts, and indirect network/install behavior.
- Tool output, error text, `.env`, private keys, or runtime metadata is accidentally persisted or returned to the model.
- Chat histories omit the assistant message containing `tool_calls`; the model loses reasoning/call continuity or the Provider rejects the next request.

### Regulatory / Compliance Context

No sector-specific regulation is assumed. Security/privacy controls follow the project's local-first contract: no credential persistence, no remote telemetry, explicit approval, bounded data exposure, and durable audit records. Canon path traversal, injection, and credential-leak controls remain subject to `$gsd-secure-phase`; this AI-SPEC adds product-specific agent behavior, not a substitute compliance claim.

### Domain Expert Roles for Evaluation

| Role | Responsibility |
|------|---------------|
| CLI maintainer | Owns deterministic fixtures, Provider wire parity, state-machine invariants, and cross-platform CI. |
| Security reviewer | Labels path/secret/destructive fixtures and audits finite process schemas and denial parity. |
| Non-programmer UAT reviewer | Judges whether approval and indeterminate-result wording are understandable and non-pressuring. |
| Provider integration reviewer | Confirms Responses/Chat history and stable call-ID mapping against official contracts. |

## 2. Framework Decision

**Selected Framework:** No external AI orchestration framework; native Rust protocol/core state machine with direct HTTP Provider adapters

**Version:** Rust 1.97.0 / edition 2024; project protocol schema v1; existing pinned Cargo dependencies

**Rationale:**

- The phase is a small, explicitly bounded single-agent loop, not a multi-agent graph or RAG system.
- The repository already has typed Provider streams, a deterministic core reducer, durable JSONL sessions, cancellation, and two direct HTTP adapters. An SDK/framework would duplicate those boundaries and make recovery semantics less auditable.
- None of the candidate frameworks provides a first-class Rust runtime that preserves this project's model-agnostic MiniMax/OpenAI-compatible contract.
- The hardest work is application-level effect safety and persistence ordering; a generic orchestration framework does not solve those constraints.
- Direct adapters preserve Provider flexibility and keep the base artifact/RSS/dependency goals credible.

**Alternatives Considered:**

| Framework | Ruled Out Because |
|-----------|------------------|
| LangGraph | Strong checkpointing, but Python/TypeScript graph state would duplicate the Rust reducer and session journal and add a second persistence model. |
| OpenAI Agents SDK | No Rust implementation, OpenAI-first abstractions, and its built-in tool/trace behavior would weaken the explicit Provider-neutral boundary. |
| the agent Agent SDK | Claude-only, MCP/shell-oriented, no Rust implementation, and outside the locked MiniMax/provider-flexible scope. |
| LangChain | Broad integration surface and abstraction overhead without value for a fixed eight-tool loop. |
| CrewAI/AutoGen/Google ADK | Multi-agent focus is out of scope; their memory/session layers would create the coupling this rewrite removes. |

**Vendor Lock-In Accepted:** No. Wire-specific request/stream shapes live only in `minimax-provider`; core/protocol/tool contracts remain Provider-neutral.

### Installation

```powershell

```

### Core Imports

```rust
use minimax_protocol::{ToolCallId, ToolDefinition, ToolInvocation, ToolResult, TurnRequest};
use minimax_core::{ApprovalPort, InvocationMachine, PermissionMode, ToolPort};
```

### Entry Point Pattern

```rust
async fn run_agent_turn(
    provider: &dyn ProviderPort,
    approvals: &dyn ApprovalPort,
    tools: &dyn ToolPort,
    store: &mut RuntimeStore,
    request: TurnRequest,
) -> Result<RunOutcome, AgentError> {
    // 1. Stream/assemble complete Provider tool calls.
    // 2. Persist each normalized request before approval.
    // 3. Apply the process-local permission policy and shared hard gates.
    // 4. Persist started, execute once, then persist one terminal result.
    // 5. Send durable tool results back through the same Provider protocol.
    // 6. Repeat within finite budgets or return the final visible response.
    RuntimeCoordinator::new(provider, approvals, tools, store)
        .run(request)
        .await
}
```

### Key Abstractions

| Concept | What It Is | When You Use It |
|---------|------------|-----------------|
| `ToolDefinition` | Provider-neutral strict name/description/JSON Schema record. | Advertise the finite v1 tool set on an agent turn. |
| `ToolInvocation` | Complete normalized call ID, name, arguments, and effect metadata. | Persist before approval and feed the core reducer. |
| `InvocationMachine` | Pure legal-transition reducer. | Reject duplicates/illegal states and emit persistence/execution effects. |
| `PermissionMode` | Exactly `confirm` or `full_access`, held in process memory. | Decide whether a known/preflight-safe call needs an approval port. |
| `ApprovalPort` | Interactive decision boundary with a deterministic headless rejection implementation. | Keep UI out of core and make approval tests injectable. |
| `ToolPort` | Async one-invocation executor returning a common terminal result. | Keep filesystem/process code out of core. |
| `RuntimeStore` | Append-only journal plus atomic derived index. | Make every invocation boundary recoverable. |

### Common Pitfalls

1. Do not execute on streamed argument deltas; wait for one complete call and validate the final JSON object.
2. Preserve the exact Provider call ID. For Chat Completions, append the full assistant message containing `tool_calls` before `role=tool`; for Responses, return `function_call_output` with the matching `call_id`.
3. Do not let `full-access` select policy, skip preflight, or persist across construction; it skips only the approval prompt.
4. Do not put arbitrary command strings behind a “bounded shell” label; use finite action enums and direct process spawning.
5. Persist the terminal result before publishing it or building the next Provider request; recovery cannot repair a result the Provider saw but the journal did not.

### Recommended Project Structure

## 4. Implementation Guidance

**Model Configuration:**
Use the session's already pinned Provider profile/model and existing credential/config chain. Keep `tool_choice=auto`; advertise only v1 function tools; do not expose Provider-hosted tools, MCP, web/file search, computer use, or code interpreter. Preserve the profile's temperature/max-output settings and bound the agent loop separately. Default tests use scripted Providers only and make no paid request.

**Core Pattern:**
A deterministic outer coordinator alternates `Provider round -> complete tool calls -> durable serial decisions/executions/results -> Provider round` until the Provider emits final text or a finite budget/cancellation/typed failure ends the run. The model proposes; Rust validates, approves, executes, records, and decides whether progress is legal.

**Tool Use:**
Generate JSON Schema from static versioned Rust definitions or hand-authored constants checked against the Rust argument types. Set strict validation where the Provider supports it, but repeat validation locally because Provider conformance is not a safety boundary. Convert every adapter outcome—including rejection, timeout, cancellation, schema failure, and indeterminate recovery—to a bounded JSON result string associated with the same call ID.

**State Management:**
Core transitions are synchronous and pure. The CLI coordinator persists effects through `RuntimeStore` before executing the next effect. The append-only journal is authoritative; indexes/checkpoints are derived. Permission mode is deliberately not state. Recovery finalizes unfinished model turns as Phase 2 already does and additionally closes unfinished tool invocations according to the pre-start/post-start uncertainty boundary.

**Context Window Strategy:**
Tool definitions are stable and sent in deterministic name order to improve request-prefix/cache stability. Tool results contain only a small structured envelope and bounded safe output; raw stderr and large files never enter context. Phase 2's deterministic compaction retains terminal tool facts/call IDs needed for continuity, not raw output bodies. For MiniMax Chat compatibility, preserve the complete assistant tool-call message required for the next round while excluding private raw reasoning from trace presentation.

## 4b. AI Systems Best Practices

### Structured Outputs with Pydantic

The production implementation is Rust/Serde. This Pydantic model is an optional dev/eval mirror for language-neutral fixture validation only; it is not a runtime dependency and never authorizes execution.

```python
from typing import Literal
from pydantic import BaseModel, ConfigDict, Field

class ToolResultFixture(BaseModel):
    model_config = ConfigDict(extra="forbid")

    schema_version: Literal[1] = 1
    call_id: str = Field(min_length=1, max_length=256)
    tool_name: Literal[
        "read_file", "list_directory", "apply_patch", "write_file",
        "run_diagnostic", "git_status", "git_diff", "npm_diagnostic",
    ]
    status: Literal[
        "succeeded", "failed", "rejected", "cancelled", "indeterminate"
    ]
    code: str = Field(min_length=1, max_length=64)
    output: str | None = Field(default=None, max_length=65536)

# Eval harness: validate once. Validation failure is a failed fixture;
# it is never retried by relaxing the schema or guessing a missing field.
```

### Async-First Design

Provider I/O and process/file adapters are async at the CLI/tools boundary; core reducers remain synchronous. Never hold a mutable journal borrow across an awaited Provider/process operation. Stream model output, but await a complete validated call before approval/execution. Cancellation must propagate into the current adapter and the outer agent budget.

### Prompt Engineering Discipline

- System/developer instructions state that tools are suggestions subject to local policy and that a rejection/indeterminate result is authoritative.
- Tool descriptions are short, factual, effect-labeled, and deterministic; they do not encourage the model to bypass approval or rephrase forbidden actions.
- The Provider receives only published schemas and bounded prior results. No few-shot example may contain credentials, absolute user paths, or a destructive workaround.
- When a tool returns `rejected`, `failed`, or `indeterminate`, instruct the model to explain the limitation or choose another allowed action—not to claim completion.

### Context Window Management

Keep stable system instructions and tool definitions at the prefix, then the compacted session summary/recent turns and native tool-call/result items. Count tool definition/result bytes in the context estimate. On compaction, preserve call ID, name, terminal status/code, and safe short facts; discard raw output already superseded by a later state.

### Cost and Latency Budget

- Phase 3 adds no model routing and no additional judge call during normal use.
- Bound model rounds and tool calls so a repeated-call loop cannot create unbounded cost.
- Offline fixtures cover the loop; no live eval runs in default CI.
- Tool definitions are deterministically ordered and unchanged across rounds where possible to preserve Provider prompt caching.
- Local preflight and persistence should remain negligible relative to Provider latency; diagnostics retain the 30-second hard timeout.

## 6. Guardrails

### Online (Real-Time)

| Guardrail | Trigger | Intervention |
|-----------|---------|--------------|
| Strict tool/schema gate | Unknown name/field, malformed/non-object/oversized arguments, duplicate ID | Block before approval; durable failed result |
| Approval gate | Confirm mode lacks an explicit matching approval | Reject before adapter; durable structured result |
| Workspace/secret gate | Absolute/escaping/protected/secret-like path or content | Block before effect; redacted denial code |
| Destructive/process gate | Non-allowlisted action, interpreter, background/network/install/Git mutation attempt | Block before spawn; no fallback shell |
| Budget/cancellation gate | Round/tool/time/output ceiling or cancellation | Stop new effects; terminate bounded child; record honest terminal/indeterminate state |
| Recovery gate | Durable `started` without durable terminal result | Record `indeterminate`; never replay automatically |
| Persistence ordering gate | Journal append/index update fails | Do not publish result or issue next Provider request |

### Offline (Flywheel)

| Metric | Sampling Strategy | Action on Degradation |
|--------|------------------|----------------------|
| Tool schema rejection rate | Local opt-in aggregate counts only; emphasize unknown-field/tool cases | Review descriptions/schema ergonomics without relaxing hard gates |
| User approval rejection rate | Local opt-in counts by tool/effect class, no arguments/output | Improve tool planning/prompt clarity; never auto-escalate permission |
| Indeterminate recovery count | Review every local occurrence | Reproduce boundary, inspect adapter cancellation/atomicity, add regression fixture |
| Repeated-call budget stops | Review every local occurrence | Add loop fixture and improve authoritative failure-result prompt wording |
| Human approval-clarity failures | UAT sample for changed prompt snapshots | Revise copy/presentation and repeat non-programmer review |

No production interaction body, tool argument, path, output, credential, or raw reasoning is uploaded for monitoring.

## 7. Production Monitoring

**Tracing Tool:** Existing local safe structured trace and append-only runtime journal (no remote telemetry; Phoenix explicitly deferred as an optional dev-only exporter)

**Key Metrics to Track:**

- Requested/approved/rejected/started/terminal invocation counts by known tool and safe status code.
- Duplicate/illegal-transition and hard-gate denial counts.
- Indeterminate recovery count (must normally remain zero).
- Tool-loop budget stops and Provider rounds per completed run.
- Adapter duration, timeout, output-overflow, cancellation, and child-cleanup outcomes without argument/output bodies.

**Alert Thresholds:**

- Any fabricated-success invariant failure, denial-parity mismatch, automatic indeterminate replay, secret fixture leak, or approval bypass fails CI immediately.
- Any real `indeterminate` recovery is prominently shown in the local session/doctor output and requires user review before claiming the task completed.
- More than one terminal result per call ID or a next Provider round before durable terminal persistence is a hard runtime error.
- Sustained p95 local preflight/persistence overhead above 50 ms (excluding file/process work) triggers profiling before release.

**Smart Sampling Strategy:**

Review all indeterminate/duplicate/secret/destructive denials; sample changed approval prompts and schema-rejection clusters; prioritize sessions with retries, budget stops, cancellations, or unusually large bounded outputs. Sampling remains local and opt-in, with sanitized structural records only.
